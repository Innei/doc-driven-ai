# doc-builder — 文档引导 Skill

> 适用于所有主流 AI agent（Claude Code、Cursor、Windsurf、GitHub Copilot 等）。  
> agent 读取本文件后，按此协议引导用户以对话方式逐节完成项目文档体系。

---

## 触发条件

用户说出以下类似语句时，激活本 skill：

- "帮我开始填文档"
- "我要用对话方式建文档"
- "doc-builder"
- "继续填 PROJECT.md / DESIGN / DEV"
- "我想从文档开始"

---

## 启动协议

### 第一步：读取上下文

按顺序读取以下文件（不存在则跳过）：

1. `AGENTS.md` — 了解项目的 AI 行为边界
2. `.doc-status.json` — 读取当前文档完成状态（不存在则视为全部未开始）
3. 任何已存在的 `[PROJECT_CODE].md`、`[PROJECT_CODE]-DESIGN.md`、`[PROJECT_CODE]-DEV.md`

### 第二步：确认项目代号

若 `.doc-status.json` 不存在，问用户：

> "你的项目代号是什么？（如 ECHO、ATLAS、POND——一个短小的全大写代号，用作所有文档文件名的前缀）"

记录代号为 `PROJECT_CODE`，并创建初始 `.doc-status.json`（见下方格式）。

### 第三步：展示状态总览

用一张简表展示当前文档完成情况，例如：

```
文档                     状态
────────────────────────────────────
MYAPP.md               ✅ reviewed（§0–§4 完成，§5–§7 待填）
MYAPP-DESIGN.md        🟡 draft（§1 完成，§2–§9 待填）
MYAPP-DEV.md           ⬜ 未开始
MYAPP-DATA.md          ⬜ 未开始（可选，按需）
```

图例：✅ reviewed / 🔒 frozen / 🟡 draft / ⬜ pending

### 第四步：让用户选择目标

问：

> "想从哪份文档继续？或者直接告诉我要填哪一节。"

若用户未明确，推荐优先级：`MYAPP.md` → `MYAPP-DESIGN.md` → `MYAPP-DEV.md` → `MYAPP-DATA.md`。

## `.doc-status.json` 规格

### 文件位置

项目根目录，建议加入 `.gitignore`（纯本地进度）或纳入版本控制（团队共享进度）——由用户决定。

### 初始结构

```json
{
  "project_code": "MYAPP",
  "created_at": "YYYY-MM-DDTHH:mm:ssZ",
  "updated_at": "YYYY-MM-DDTHH:mm:ssZ",
  "documents": {
    "MYAPP.md": {
      "status": "pending",
      "sections": {}
    },
    "MYAPP-DESIGN.md": {
      "status": "pending",
      "sections": {}
    },
    "MYAPP-DEV.md": {
      "status": "pending",
      "sections": {}
    },
    "MYAPP-DATA.md": {
      "status": "pending",
      "sections": {}
    }
  }
}
```

### 状态值

| 值 | 含义 |
|----|------|
| `pending` | 未开始 |
| `draft` | 已写入文件，未经质量审核 |
| `reviewed` | 通过质量检查 |
| `frozen` | 用户显式锁定，不再修改 |

### 聚合规则（文档级 status）

- 所有章节均为 `reviewed` → 文档为 `reviewed`
- 任意章节为 `frozen` 且其余 `reviewed` → 文档为 `frozen`
- 否则取最低完成度

### 更新时机

每次成功写入一个章节后，立即更新 `.doc-status.json` 中对应章节状态为 `draft`，并更新 `updated_at`。

## 问答引导协议

### 通用规则

1. **一次一问**：每次只问一个问题，不堆问题。
2. **写入确认**：生成章节草稿后，展示给用户，得到确认再写入文件。
3. **草稿格式**：完整引用模板占位符位置，将用户回答填入，保留原有结构与章节编号。
4. **跳过选项**：若某章节模板标注"可选"，告知用户可跳过。
5. **断点续写**：若 `.doc-status.json` 显示某章节已完成，告知用户并询问是否重新填写。

### 引导节奏示例

```
agent: "§0 要描述现有方案的问题。请告诉我：用户现在通常用什么方法来解决这个问题？（一两句话，不需要完整）"

用户: "他们用 Figma 手动调色盘"

agent: "好。这个方法最核心的缺陷是什么？（比如：步骤太多、结果不可直接用、需要审美经验等）"

用户: "每次都要从零开始，没有可复用的数学约束"

agent: "明白了。基于这个问题，你的产品有哪些核心假设？（可以说 1–3 条，例如：'用户已经知道颜色目标，产品不需要教'）"

...（收集完毕后）

agent: "根据你的回答，§0 草稿如下：

---
## 0. 这个产品要解决的问题

市面上的主流解法是 Figma 手动调色。它的核心问题是：每次都要从零开始，没有可复用的数学约束。

本产品的核心假设：
1. **用户已知颜色目标** — 产品的工作是不挡路
2. **有效配色可以数学化** — 色阶、对比度约束可量化
---

这个草稿合适吗？（直接说'确认'或告诉我要修改哪里）"
```

## 各文档问答脚本

---

### 文档一：`[PROJECT_CODE].md`（产品意图）

对应模板：`docs/templates/PROJECT.md`

#### §0 — 核心问题与假设

**Q1**：用户现在通常用什么方法（竞品/现状）来解决这个问题？列出 1–3 个现有方案。

**Q2**：这些方案共同的核心缺陷是什么？（要求可量化或可验证，拒绝"不够好用"这类形容词）

**Q3**：你的产品建立在哪些核心假设上？（1–3 条，每条一句话，格式：假设 → 一句解释）

#### §1 — 一句话定位

**Q1**：用一句话定义这个产品："`[PROJECT_CODE]` 是一个 ___"

**Q2**：它**不是**什么？（列出 1–2 个容易混淆的产品形态）

#### §2 — 核心体验

**Q1**（§2.1）：用户打开产品的第一秒，看到什么？这一刻的设计目标是什么？

**Q2**（§2.2）：整个产品**只有一个**真正的核心交互——那个交互是什么？

**Q3**（§2.3）：用户完成核心流程后，会得到什么？这个产出的成功标准是什么？

#### §3 — 用户旅程

**Q1**：用户从入口到结果的完整路径是什么？（格式：入口 → 步骤1 → 步骤2 → 结果）

**Q2**：每一步用户看到什么、做什么？有哪些异常或岔路？（可用表格或分点说明）

#### §4 — 领域模型

**Q1**：这个产品的核心领域实体有哪些？（1–3 个，每个给一个名字和一句定义）

**Q2**（可跳过）：实体之间有什么依赖或组合关系？（简单项目可跳过）

#### §5 — 品牌声音（可选）

告知用户：此节适合内容类/情感类产品，工具类可跳过。

若继续：**Q1**：产品说话的语调是什么？（2–3 个特征词，但必须能转化为可执行规则）

**Q2**：写作禁忌是什么？（至少 2 条 ❌ 禁止项）

**Q3**：使用第几人称？

#### §6 — 产品禁忌（必填）

**Q1**：有哪些功能或设计方向**明确不做**？（至少 3 条，格式：🚫 + 具体描述）

> 质量要求：每条必须够具体，能直接作为 AI 的护栏。"不要太复杂"不是禁忌，"不引入用户账户体系"才是。

#### §7 — 隐私与数据原则

**Q1**：用户数据存在哪里？（仅本地 / 加密后服务器 / 不存储）

**Q2**：有哪些硬性约束？（不采集行为数据 / 不需要账户 / 等）

### 文档二：`[PROJECT_CODE]-DESIGN.md`（视觉契约）

对应模板：`docs/templates/PROJECT-DESIGN.md`

> 注意：DESIGN 文档全英文写作。agent 收集中文回答后，在填写时自动翻译为英文。

#### §1 — 核心隐喻

**Q1**：用一个比喻描述这个产品的整体视觉感——它的界面像什么？（如："paper and ink"、"deep-night radio"、"archive cabinet"）

**Q2**：用 2–3 句展开这个比喻：它如何让调色板、字体、动效规则感觉"理所当然"而不是任意的？

#### §2 — 调色板

**Q1**（§2.1）：颜色系统的"物质逻辑"是什么？（不是色值，而是为什么这些颜色是对的——如："ink-on-paper，不用主色，不用语义色"）

**Q2**（§2.2）：日间模式的 token 表（按模板格式列出 `--paper`、`--ink-100` 等，含 hex 值和用途说明）

> 若用户暂无色值，可先填用途说明，hex 留空，标记为 `draft`。

**Q3**（§2.3）：是否有夜间模式？若有，给出 night face token 表；若无，说明原因。

**Q4**（§2.4）：禁止使用的颜色或语义（如：禁止引入 `--color-primary`）

#### §3 — Typography

**Q1**：字号梯度有几档？每档的用途是什么？（格式：`xs 12px body`、`sm 14px meta`……）

**Q2**：字体选型是什么？（正文、代码、展示型各用什么）

**Q3**：有哪些排版禁忌？（如："禁止超过 6 档字号"、"禁止 font-weight: 300"）

#### §4 — Spacing（间距系统）

**Q1**：间距系统基于什么基数？（如：4px 倍数、8px 倍数）

**Q2**：常用间距档位有哪些？（如：4/8/12/16/24/32/48px）

#### §5 — Layout（布局）

**Q1**：最大宽度是多少？（如：672px、1024px、无限制）

**Q2**：有哪些布局约束？（如：单栏、两栏最大、不允许全宽卡片）

#### §6 — Components（组件边界）

**Q1**：哪些组件类型**禁止存在**？（如：Modal / Toast / Sidebar / 全局 Nav）

**Q2**：哪些组件是允许的核心组件？（列出 3–5 个核心组件类型）

#### §7 — Motion（动效规则）

**Q1**：动效只允许出现在哪里？（如：只有内容入场时有过渡，其余全部 snap）

**Q2**：具体的动效参数约束是什么？（如：duration ≤ 200ms、easing: ease-out、禁止弹性动画）

#### §8 — Accessibility

**Q1**：前景/背景对比度是否满足 WCAG AA（最低 4.5:1）？颜色 token 是否已验证？

**Q2**：`prefers-reduced-motion` 时系统如何响应？（如：所有 spring 动画退化为 instant，不隐藏内容）

> 若 §2 的 night face 已处理对比度，此节可简短。

#### §9 — Enforcement（约束落地）

**Q1**：这些视觉约束如何在工程中强制执行？（如：所有 token 通过 design token 文件引用，lint 规则禁止 hex 字面量）

---

### 文档三：`[PROJECT_CODE]-DEV.md`（工程规范）

对应模板：`docs/templates/PROJECT-DEV.md`

#### §一.1 — 技术栈

**Q1**：填写技术栈表（UI 框架、构建工具、路由、状态管理、类型系统、样式方案、包管理器、部署平台）。

> 若用户不确定样式方案，提示：项目推荐使用 vanilla-extract（与 DESIGN 文档的 token 系统配合最好）。

#### §一.2 — 渲染策略

**Q1**：产品整体的渲染策略是什么？（SSG / SSR / SPA / 混合）

**Q2**：每条路由用哪种渲染方式？为什么？（按模板格式列表）

#### §一.3 — 状态管理

**Q1**：产品有哪些独立的状态领域？（如：主题、流程进度、结果、历史）

**Q2**：哪些状态需要持久化（写 localStorage）？哪些绝对不能持久化？

**Q3**：每个 store 的职责、服务范围、持久化方式各是什么？

#### §一.4 — 路由与目录结构

**Q1**：路由配置方式是什么？（file-based / 手写路由表 / 框架约定）

**Q2**：`app/` 目录结构是什么？（可直接采用模板默认结构并说明修改点）

#### §二.1 — 核心类型定义

**Q1**：产品的数据分为哪几层？（静态数据层 / 运行时状态层 / 结果输出层 / 持久化层）

**Q2**：每层的核心字段是什么？（不需要完整类型，关键字段即可）

#### §三 — 编码/序列化（可选）

**Q1**（前置）：产品是否有分享链接或可持久化的用户数据？若无，跳过本节。

**Q2**：内部数据如何序列化为可分享字符串？（明文格式是什么）

**Q3**：编码算法步骤？解码失败如何处理？

#### §四 — 冻结条款（必填，若有持久化数据）

**Q1**：哪些数据格式或常量，一旦上线就不可更改？（列出 1–N 条，每条说明为何不可更改）

> 质量要求：冻结条款不能是"功能不变"，必须是"具体格式/值/顺序/字符集"级别的约束。

#### §六 — 错误处理与降级

**Q1**：错误呈现的风格是什么？（符合 DESIGN 文档的审美约束）

**Q2**：哪些错误静默处理、不提示用户？

### 文档四：`[PROJECT_CODE]-DATA.md`（结构化数据，可选）

对应模板：`docs/templates/PROJECT-DATA.md`

**前置判断**：若产品不含结构化数据（题库、配置项、映射表等），告知用户此文档可跳过。

#### 格式说明节

**Q1**：本文档使用哪些标记符号？每个标记含义是什么？（可沿用模板默认标记或自定义）

#### 数据条目节

按用户提供的内容逐条录入，agent 负责格式化为模板格式（带 ID、带标记、带分隔线）。

## 质量检查规则

每个章节在标记为 `reviewed` 前，agent 须检查以下项：

### 通用规则

| 检查项 | 不合格示例 | 合格示例 |
|--------|-----------|---------|
| 无占位符 | `[项目代号]`、`TBD`、`TODO` 仍存在 | 已替换为真实内容 |
| 无纯形容词约束 | "界面简洁优雅" | "字号只有 5 档，没有 display 尺寸" |
| 禁止列表够具体 | 🚫 "不要太复杂" | 🚫 "不引入用户账户体系" |
| 冻结条款有具体格式 | "编码不变" | "Base62 字符集固定为 0-9A-Za-z，不可增删" |

### PROJECT.md 专项

- §6（产品禁忌）：至少 3 条，每条可直接作为 AI 代码生成的 "禁止" 指令
- §2.2（核心交互）：必须是单数——不允许写"两个核心交互"

### PROJECT-DESIGN.md 专项

- §2（调色板）：若 hex 留空，标为 `draft`，提示用户补充色值后才能升为 `reviewed`
- §7（动效）：必须说明"哪里可以动"和"哪里必须 snap"，两个都要有

### PROJECT-DEV.md 专项

- §四（冻结条款）：若产品有持久化数据，此节不可跳过
- §一.3（状态管理）：必须明确哪些 store 持久化、哪些不能

## 写入协议

### 首次创建文档

若目标文档不存在，从 `docs/templates/` 复制对应模板，重命名为 `[PROJECT_CODE]-*.md`（或直接创建于项目根目录），再填入内容。

### 章节写入

只修改对应章节（`## X` 至下一个 `## X+1` 之间的内容），不覆盖其他已完成章节。

### 更新状态

每次成功写入后：
1. 将 `.doc-status.json` 中对应章节状态更新为 `draft`
2. 更新文档级 `status`（按聚合规则）
3. 更新 `updated_at`

### 质量审核后

通过所有检查项后，将章节状态更新为 `reviewed`。

### 冻结

用户明确说"锁定 §X"或"freeze §X"时，将对应章节状态更新为 `frozen`，并在文档内该章节顶部加注：

```
<!-- FROZEN: YYYY-MM-DD — 此节已锁定，修改需用户显式解冻 -->
```

## 会话结束协议

每次引导结束时（用户说"先到这里"或会话中断前），agent 须：

1. 展示本次会话更新的章节列表（`draft` / `reviewed` 各几节）
2. 展示剩余待填章节的简表
3. 提示用户：下次继续时，告诉 agent "doc-builder 继续"，agent 会读取 `.doc-status.json` 从断点恢复

示例：

```
本次完成：MYAPP.md §0（draft）、§1（reviewed）
待填节点：MYAPP.md §2–§7、MYAPP-DESIGN.md 全部

下次说「doc-builder 继续」即可从 §2 接续。
```

## 跨 agent 兼容说明

本 skill 以纯 Markdown 编写，无任何 agent 专属 API 调用。

所有状态通过读写 `.doc-status.json` 文件持久化。  
所有文档通过读写 `[PROJECT_CODE]-*.md` 文件持久化。  
agent 只需具备"读文件 + 写文件 + 对话"能力即可执行本 skill。
