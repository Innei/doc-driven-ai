# 工作流：文档驱动开发的最佳实践

> 从"有一个想法"到"AI 生成可运行的代码"，文档驱动开发的典型工作流是什么？

---

## 工作流概览

```
意图 → 契约 → 规范 → 数据 → 原型 → 迭代 → 冻结
```

| 阶段 | 输出物 | 负责方 |
|------|--------|--------|
| 意图 | `PROJECT.md` 初稿 | 人类（产品/设计） |
| 契约 | `PROJECT-DESIGN.md` 核心章节 | 人类（设计） |
| 规范 | `PROJECT-DEV.md` 技术底座 | 人类 + AI |
| 数据 | `PROJECT-DATA.md` / `PROJECT-[TOPIC].md` | 人类（内容） |
| 原型 | 第一个可运行的代码提交 | AI |
| 迭代 | 文档修正 + 代码补全 | 人类 + AI |
| 冻结 | `PROJECT-DEV.md` 冻结条款锁定 | 人类 |

---

## 阶段详解

### 阶段一：意图（Intent）

**目标**：用最短的文档回答"这是什么"和"它拒绝什么"。

**做法**：
- 写 `PROJECT.md` 的 §0（核心假设）和 §1（一句话定位）
- 写 §2（核心体验）——用户打开产品的第一秒看到什么、最核心的一个交互是什么
- 写 §6（产品禁忌）——这一步极其重要，它是 AI 的"护栏"

**时长**：15–30 分钟。不要追求完美，先有一个"足够好"的初稿。

---

### 阶段二：契约（Contract）

**目标**：把意图转译为不可违反的视觉约束。

**做法**：
- 写 `PROJECT-DESIGN.md` 的 §1（设计信条和核心隐喻）
- 确定 §2（Color）和 §3（Typography）
- 写 §9（Enforcement）——如何让这些约束在工程中被强制执行

**检查点**：设计文档里的每一条规则都应该能被 AI 直接翻译成代码或 lint 规则。形容词不是规则，可量化的约束才是。

---

### 阶段三：规范（Spec）

**目标**：选择技术栈并定义工程边界。

**做法**：
- 写 `PROJECT-DEV.md` 的 §一（技术栈 + 渲染策略 + 状态管理）
- 定义 §二（核心数据结构）
- 如果产品有分享链接或持久化格式，先写 §四（冻结条款），再写 §三（编码算法）

**AI 的角色**：把 `PROJECT.md` 和 `PROJECT-DESIGN.md` 喂给 AI，让它推荐技术栈方案，然后基于推荐写出 `PROJECT-DEV.md` 的正式版本。

---

### 阶段四：数据（Data）

**目标**：把内容资产从代码中剥离出来。

**做法**：
- 如果有结构化数据（题库、配置项、映射表），写入 `PROJECT-DATA.md`
- 如果有专题内容（文案、独白、提示词、算法说明），按领域拆分写入多份 `PROJECT-[TOPIC].md`
- 用清晰的 Markdown 结构组织，方便后续解析为 JSON/TS

**内容文档拆分参考**：

| 文档 | 内容类型 |
|------|---------|
| `PROJECT-DATA.md` | 结构化数据：带 ID、带标记、需要机器解析的条目 |
| `PROJECT-QUESTIONS.md` | 题库、问卷、交互流程节点 |
| `PROJECT-COPY.md` | UI 短文案、错误提示、按钮标签 |
| `PROJECT-MONOLOGUES.md` | 长散文文案、结果页独白 |
| `PROJECT-PROMPTS.md` | AI 生图/生文提示词 |
| `PROJECT-SCORING.md` | 算法逻辑、评分依据、类型映射 |

并非所有项目都需要所有这些文档。按实际需要取用，没有的类型直接忽略。

**关键原则**：非技术人员应该能直接编辑这些 `.md` 文件，而不需要看懂代码。

---

### 阶段五：原型（Prototype）

**目标**：让 AI 基于已有文档生成第一个可运行的版本。

**指令示例**：

> "请基于 `MYAPP-DEV.md` §一.1 的技术栈和 §一.4 的目录结构，初始化项目。然后基于 `MYAPP-DESIGN.md` §2–§3 创建 `app/styles/tokens.css.ts`，基于 `MYAPP-DATA.md` 创建 `app/lib/data/items.ts`。最后创建 `app/routes/_index.tsx` 作为首页。"

**建议**：把原型生成拆成 2–3 个小任务，而不是一次性让 AI 写完整项目。这样更容易在每个步骤中基于文档检查输出。

---

### 阶段六：迭代（Iterate）

**目标**：在实践中修正文档和代码。

**常见迭代模式**：

1. **设计文档太宽松** → 发现 AI 产出了你不想要的 UI → 回到 `PROJECT-DESIGN.md` 增加一条禁止规则
2. **数据结构缺失字段** → 发现某个交互状态无法表达 → 回到 `PROJECT-DEV.md` §二 更新类型定义
3. **文案需要调整** → 直接编辑对应的 `PROJECT-[TOPIC].md` → 让 AI 同步更新代码中的引用

**文档永远先于代码更新**。如果你发现代码需要某种 hack 才能工作，那通常是文档没有描述清楚边界。

---

### 阶段七：冻结（Freeze）

**目标**：在产生用户数据之前，锁定向后不兼容的决策。

**做法**：
- 审查 `PROJECT-DEV.md` 的"冻结条款"章节
- 确认哪些改动会导致旧链接/旧存档失效
- 把这些条款写成不可协商的硬性规则

**时机**：通常在项目上线前 1–2 周，或第一次有用户分享链接/生成可持久化数据时。

---

## 协作模式：人类与 AI 的分工

### 人类最擅长的

- **意图判断**：这个产品应该拒绝什么？核心交互是否足够简单？
- **审美决策**：这条规则是否过于严格？那个颜色是否合适？
- **内容创作**：文案、题目、提示词
- **冻结裁决**：哪些格式一旦确定就不能改

### AI 最擅长的

- **技术翻译**：把设计文档的约束写成 token 和组件
- **数据解析**：把 Markdown 内容文档转换为结构化代码
- **一致性检查**：扫描代码是否违反了文档中的禁止列表
- **样板代码**：路由、store、构建配置、SEO 元数据

## 一个实用的 daily workflow

**上午**：
1. 花 10 分钟审阅昨天的代码，找出与文档不一致的地方
2. 更新对应的 `PROJECT-*.md` 文档
3. 给 AI 一个基于文档的明确任务（如："实现 `PROJECT-DESIGN.md` §7 的入场动画"）

**下午**：
4. 审查 AI 产出，对照文档检查
5. 如果发现问题，指出具体违反的章节号，让 AI 修正
6. 如果文档本身有问题，先改文档，再让 AI 跟上来

**每周五**：
7. 做一轮"文档漂移检查"：代码里是否有新引入的颜色字面量？是否有未在 DESIGN 文档中声明的组件？
8. 更新冻结条款，标记本周已锁定的决策

---

## 长期维护

文档驱动项目不是"写一次就忘"。随着时间推移：

- **产品进化** → 更新 `PROJECT.md` → 级联到 DESIGN 和 DEV
- **设计迭代** → 更新 `PROJECT-DESIGN.md` → 级联到 DEV 和组件
- **技术重构** → 更新 `PROJECT-DEV.md` → 确保新的实现仍满足 DESIGN 约束
- **内容变更** → 更新对应的 `PROJECT-[TOPIC].md` → AI 同步代码引用

把 `PROJECT-*.md` 当作你的**架构决策记录（ADR）**和**设计系统文档**的合体。它们值得花时间维护。
