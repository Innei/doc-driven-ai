# 使用指南：如何启动一个文档驱动项目

> 这份指南假设你已经认同"文档即模板"的理念，并准备用 `ai-starter-kit` 开始一个新项目。

---

## 第一步：初始化仓库

在你的新目录中，只做两件事：

1. **创建 `AGENTS.md`**

   从 `ai-starter-kit` 复制 `AGENTS.md` 到你的仓库根目录。这份文件是 AI 代理进入项目后的第一入口。

2. **确定项目代号**

   选一个简短的代号（如 `ECHO`、`ATLAS`、`POND`），所有核心文档都以这个代号为前缀。

---

## 第二步：三份核心文档的填空顺序

### 2.1 先写 `PROJECT.md`（产品意图）

不要先选技术栈。先回答这些问题：

- 用户打开这个链接的第一秒，会看到什么？
- 用户会做的第一个动作是什么？
- 这个动作做完之后，用户会得到什么？
- **最重要的是**：它**拒绝**长成什么样？

`PROJECT.md` 里的"产品禁忌"往往比功能列表更重要。它们是 AI 在后续开发中最需要的上下文。

### 2.2 再写 `PROJECT-DESIGN.md`（视觉契约）

基于 `PROJECT.md` 的意图，锁定审美的"不可协商项"：

- 调色板有几档？为什么不能有第七档？
- 动画只允许出现在哪里？其他地方必须 snap（瞬间切换）？
- 哪些组件类型**禁止存在**？

DESIGN 文档的核心目标不是"提供选择"，而是"移除坏选项"。每一条规则都应该让未来的选择变得更少、更简单。

### 2.3 最后写 `PROJECT-DEV.md`（工程规范）

现在你才需要考虑技术实现。把 DESIGN 文档的约束翻译成工程决策：

- 为了支持"首屏无白屏"，哪些路由走 SSG？
- 为了支持"零运行时样式注入"，是否选择 vanilla-extract / Linaria / 纯 CSS？
- 状态如何拆分？哪些可以 persist，哪些绝对不能？
- 用户数据（如分享链接）的编码/解码规则是什么？
- **冻结条款**：一旦上线后改动会导致旧链接失效的数据格式和常量。

---

## 第三步：按需拆分内容文档

如果你的项目包含大量文案、题目、配置项，不要把它们塞进代码里。按领域拆分为独立的 `PROJECT-[TOPIC].md`：

| 文档 | 内容 | 示例 |
|------|------|------|
| `PROJECT-DATA.md` | 结构化数据，带 ID 和标记，供脚本解析 | 题库、映射表、配置项 |
| `PROJECT-[TOPIC].md` | 专题内容资产，按领域命名 | `PROJECT-COPY.md`、`PROJECT-PROMPTS.md`、`PROJECT-SCORING.md` |

不是所有项目都需要这些文档。工具类、效率类产品可能只需要三份核心文档就够了。

---

## 第四步：与 AI 协作开发

当你对 AI 发出指令时，尽可能**引用文档章节**。好的指令看起来像：

> "根据 `MYAPP-DEV.md §一.1`，项目使用 React 19 + vanilla-extract。请基于 `MYAPP-DESIGN.md §2.2` 的调色板和 `§3.1` 的字体栈，在 `app/styles/tokens.css.ts` 中创建 design token 契约。注意 `DESIGN §2.4` 的禁止列表：不要引入 `--color-primary`。"

这种指令的好处：
- AI 不需要"猜测"你的偏好
- 产出的代码与文档保持一致
- 代码注释中自然可以带上文档引用

---

## 第五步：维护文档与代码的同步

文档驱动开发最大的风险是**文档与代码漂移**。当你或 AI 修改了代码中的关键行为时，必须同步更新文档。

### 需要同步文档的场景

| 代码变更 | 需要同步的文档 |
|----------|---------------|
| 新增了一个颜色变量 | `PROJECT-DESIGN.md §2` |
| 改了路由渲染策略 | `PROJECT-DEV.md §一.2` |
| 改了状态持久化边界 | `PROJECT-DEV.md §一.3` |
| 改了分享链接的编码格式 | `PROJECT-DEV.md §三` + 冻结条款 |
| 新增了一个组件类型 | `PROJECT-DESIGN.md §6` |
| 修改了内容文案 | 对应的 `PROJECT-[TOPIC].md` |

建议在项目的 PR 模板中加一条：

> "如果本次改动触及设计 token、数据结构、路由策略或冻结常量，请同时更新对应的 `PROJECT-*.md` 文档。"

---

## 常见错误

### ❌ 把 README 当产品文档

README 是给人类的快速开始指南。`PROJECT.md` 是给 AI（和人类）的产品意图契约。两者可以共存，但功能不同。

### ❌ 设计文档写得太抽象

"简洁优雅"不是约束。"字号梯度只有六档，没有 text-xl"才是约束。AI 需要的是可执行的规则，不是形容词。

### ❌ 忘记写冻结条款

任何会产生用户数据持久化的项目（URL、localStorage、分享链接）都必须有冻结条款。否则六个月后你会想"改一下格式"，然后发现所有旧链接都失效了。

### ❌ 把所有内容都堆进一份文档

当 `PROJECT-DATA.md` 超过 500 行，就该按领域拆分了。一份文档一个关注点：结构化数据一份，长文案一份，算法逻辑一份。

### ❌ 一次性写完全部文档再开始编码

文档驱动不意味着"瀑布式文档"。你可以先写 `PROJECT.md` 的初稿和 `PROJECT-DESIGN.md` 的核心三章，就开始让 AI 生成第一个原型。在实践中迭代文档，比在文档阶段空想更有效。
