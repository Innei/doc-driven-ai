# AGENTS.md — AI 上下文协议

> 这份文档是给 AI 的"使用说明书"。它规定了 AI 在阅读项目文档、生成代码、审查修改时应该遵循的元规则。
>
> **任何进入本仓库的 AI 代理，都应首先阅读本文件，再阅读 `PROJECT.md` 及其衍生文档。**

---

## 1. 文档阅读顺序

当你开始处理这个项目的任何任务时，按以下顺序读取文档：

1. **`AGENTS.md`**（本文件）— 理解 AI 的行为边界
2. **`PROJECT.md`** — 理解产品意图
3. **`PROJECT-DESIGN.md`** — 理解视觉与交互约束
4. **`PROJECT-DEV.md`** — 理解工程规范与数据结构
5. **相关的 `PROJECT-*.md` 数据文档** — 获取具体的内容资产

**不要跳过前面的文档直接读开发文档。** 意图和契约层是开发层的上游。

---

## 2. 文档冲突时的裁决规则

如果不同文档对同一件事给出了不同指令，优先级如下：

```
PROJECT.md（意图） > PROJECT-DESIGN.md（视觉契约） > PROJECT-DEV.md（实现草稿）
```

- 如果 `PROJECT-DEV.md` 的某项技术实现与 `PROJECT-DESIGN.md` 的审美约束冲突，**以 DESIGN 文档为准**，并提示用户需要更新 DEV 文档。
- 如果 `PROJECT-DESIGN.md` 的某个视觉决策与 `PROJECT.md` 的产品意图冲突，**以 PROJECT.md 为准**，并提示用户需要重新评估设计方向。

---

## 3. 代码生成原则

### 3.1 先查文档，再写代码

在生成任何组件、样式、数据结构或路由之前，先确认：
- 这个功能在文档里有没有被描述过？
- 对应的章节号是什么？
- 有没有相关的"禁止列表"或"冻结条款"？

### 3.2 从文档引用到代码注释

当你基于文档的某个具体规则生成代码时，**在代码注释中引用文档来源**。例如：

```tsx
// 动画参数来自 DESIGN §7.2
const springs = { enter: { stiffness: 100, damping: 18, mass: 1 } };
```

这样做有两个好处：
1. 人类审查者可以追溯到原始决策
2. 未来的 AI 代理在修改这段代码时知道它服务于哪条契约

### 3.3 禁止列表是绝对红线

如果 `PROJECT-DESIGN.md` 或 `PROJECT-DEV.md` 中列出了"禁止存在的组件"、"禁止使用的颜色"、"禁止引入的依赖"，**无论用户如何要求，都不应违反**。如果用户明确要求添加被禁止的内容，你应该：
- 指出这是文档中的禁止项
- 解释为什么被禁止
- 提供符合文档约束的替代方案

### 3.4 数据文档优先于硬编码

项目中的题目、文案、提示词等内容资产应存放在独立的 `PROJECT-*.md` 文档中。生成代码时：
- 优先假设这些数据存在于独立的文档或从文档解析出的数据文件中
- 不要把大量文案直接嵌入 JSX/TSX
- 如果数据文档尚未被解析为结构化格式，可以建议用户创建解析脚本

---

## 4. 修改建议原则

### 4.1 文档先行

当你发现现有代码与文档不一致时，**首先指出文档中的哪一条规则被违反了**，而不是直接修改代码。除非用户明确要求你修复代码。

### 4.2 冻结条款需要显式批准

如果某条修改会触及 `PROJECT-DEV.md` 中的"冻结条款"（如改变数据格式、编码算法、URL 结构），你必须：
- 明确警告用户这会导致向后不兼容
- 说明哪些旧数据/旧链接会失效
- 建议用户先在文档中更新冻结条款，再修改实现

---

## 5. 提问策略

当你需要更多信息才能继续时，尽量基于项目已有的文档框架来提问。例如：

> ❌ 不好的提问："你想用什么颜色？"
> ✅ 好的提问："根据 DESIGN §2 的调色板，你打算新增一个 token 还是复用现有的 `--ink-60`？"

> ❌ 不好的提问："这个页面要不要做动画？"
> ✅ 好的提问："根据 DESIGN §7.1 的两层动效规则，这个交互属于 Tier 1（内容入场）还是 Tier 2（状态切换应 snap）？"

---

## 6. 文件操作边界

- **可以读写** `PROJECT-*.md` 文档、源代码、配置文件
- **不应修改** 已经产生用户数据依赖的冻结常量（如编码映射表、题目顺序、URL 路径）
- **必须更新** 当你修改了 `AGENTS.md` 自身引用的规则时

---

## 7. 文档维护责任

如果你：
- 改变了技术栈
- 改变了设计 token 结构
- 改变了状态管理或数据流
- 引入了新的工程约束

**你有责任建议用户同步更新对应的 `PROJECT-*.md` 文档。** 文档和代码漂移是这个模式最大的风险。
